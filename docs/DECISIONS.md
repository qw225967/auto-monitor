# Architecture Decisions

用于记录关键决策，避免后续重复争论与上下文丢失。

## ADR-001: 先保障配额与稳定性，再做机会质量升级

- 日期：2026-03-10
- 状态：accepted

背景：
- CoinGecko 有月度请求限额，且当前流程存在全表扫描与失败重试不足问题。
- 若不先做稳定性治理，后续净收益模型和执行层优化将建立在不稳定数据上。

决策：
- 优先推进 M1（配额安全和稳定性）再推进 M2/M3。
- M1 包含 header 统一、链级增量、重试退避、缓存防抖空、并发上限。

影响：
- 短期内先投入基础设施改造，业务功能迭代速度略降。
- 中长期提升系统可持续运行能力与优化迭代效率。

备选方案：
- 直接做净收益模型与执行层。未采纳：会放大数据质量问题。

## ADR-002: CoinGecko 请求从“全表定时”转为“候选优先”

- 日期：2026-03-10
- 状态：proposed

背景：
- 全表流动性同步对限额不友好，且大量请求价值低。

决策：
- 引入预算管理、负缓存、价值评分和分层 TTL。
- 候选集内高价值键优先刷新，低价值键后台低频补全。

影响：
- 请求量和无效请求占比下降。
- 需要新增元数据与调度逻辑，复杂度上升。

待确认项：
- 高/中/低价值的阈值和 TTL 参数。
- 预算触发降级策略（例如只保 top N）。

## ADR-003: 机会排序以净收益为核心，不再只看 spread

- 日期：2026-03-10
- 状态：proposed

背景：
- spread 高不代表可盈利，需扣除交易/提现/桥接/滑点/时间风险。

决策：
- 增加净收益字段并以 `net_spread` 为主排序。
- 同时给出 `confidence`（成功率与时延稳定性）。

影响：
- 候选质量提升，误报下降。
- 需要接入费率与历史统计数据。
